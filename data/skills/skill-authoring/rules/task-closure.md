# Task Closure Protocol（创作台专用）

每次生成或修改产物后，AI **必须**完成以下协议，不得跳过。

---

## 1. 30 秒 AAR（After Action Review）

产物写入完成后，在 render_result success 之前，先内部回答：

| 问题 | 要回答的内容 |
|------|------------|
| Did it work? | 产物结构是否完整？所有文件都写入了吗？ |
| Gaps? | 用户的需求是否有没覆盖到的场景？ |
| Quality? | rules/ 是约束还是步骤？workflow 是推理还是脚本？ |
| Next risk? | 用户拿去跑时最可能遇到什么问题？ |

AAR 发现问题 → 立即修复，不要留给用户踩坑。

---

## 2. Rationalizations 借口表（遇到以下想法立即停下）

以下是 AI 常见的自我合理化借口，一旦出现这些念头，视为 Red Flag：

| 借口 | 实际问题 |
|------|---------|
| "用户没有明确说要…" | 用户描述了目标，实现细节是 AI 的职责 |
| "差不多能用就行" | 产物质量标准不是"能用"，是"稳定可维护" |
| "这个规则有点严，跳过一次" | rules/ 是硬约束，没有"这次例外" |
| "workflow 写得详细点没关系" | 一旦变成步骤列表就是 Playbook，不是 Skill |
| "先生成再让用户改" | 不完整的产物比没有更危险，会被当作正确的使用 |

---

## 3. Red Flags 硬停（以下情况必须停下来说明，不得继续）

- [ ] 我不确定产物路径是否正确（`data/skills/` vs `data/playbooks/`）
- [ ] workflow 文件里出现了 3 条以上连续编号步骤
- [ ] rules/ 里写的是执行步骤而不是约束
- [ ] SKILL.md 超过 100 行
- [ ] 用户要求的功能需要破坏性操作但 rules/ 里没有确认约束
- [ ] 生成的产物覆盖了已有同名文件但没有 ask_user 确认

遇到 Red Flag → 用 `render_result format=message level=warn` 告知用户，说明问题，等待指示。

---

## 4. 泛化规则检查

写入任何 rules/ 内容前，确认：
**"把这条规则的产物名改掉，它还成立吗？"**

- "删除容器前必须确认" → 对任何删除操作都成立 ✅ 泛化
- "删除 vps-website-delete 的容器前必须确认" → 项目特定 ❌ 太窄
